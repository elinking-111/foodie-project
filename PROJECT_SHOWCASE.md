# Foodie Project Showcase

这份文档用于展示一件事：

不是先有产品团队、设计稿、技术方案，才有网站。

相反，这个项目是从几句非常自然的中文指令开始，一步步长成一个可上线、可评论、可点赞、可封装成 skill 的真实网站。

## 1. 这个项目最后做成了什么

- 一个可公开访问的美食/旅行个人网站
- 支持按地区、分类、区域筛选
- 支持地图展示
- 支持每家店单独评论
- 支持共享点赞
- 评论需要 Google 登录
- 点赞不需要手动登录，使用 Firebase Anonymous Auth
- 整套能力已经可以沉淀成 skill

## 2. 真实推进项目时，用户实际说过的关键 Prompt

下面这些不是“事后美化”的文案，而是这个项目里真正推动产出的指令型 Prompt。

### 阶段 A：从内容到网站

这类 prompt 已经沉淀在项目的评测样例里，代表最早的核心需求。

#### Prompt 1：把手机备忘录做成网站

> 我在手机备忘录里记了一堆北京和旅游的吃喝玩乐清单，帮我做成一个网站。网站叫「小王の探店日记」。这是我的笔记内容：...

这个 prompt 对应的能力是：

- 从备忘录/清单中抽取地点
- 自动判断分类
- 自动识别去过/想去/推荐
- 生成网站数据
- 生成可浏览的网站页面

#### Prompt 2：英文版城市指南

> I have a list of my favorite spots in NYC and some travel destinations. Can you turn this into a website? Call it "Alex's City Guide". Here's my list: ...

这个 prompt 对应的能力是：

- 同一套系统支持英文输入
- 自动切换网站语言
- 自动适配地区、地图中心、分类标签

#### Prompt 3：格式很随意的清单也能转网站

> 帮我把这个文件里的内容做成一个美食地图网站吧，就叫「吃货地图」。这是内容，格式比较随意：...

这个 prompt 对应的能力是：

- 不要求用户先整理结构
- 支持半结构化文本
- 依然能抽取地点、描述、标签、状态

#### Prompt 4：从小红书文案做网站

> 我从小红书上复制了几篇探店笔记的内容，帮我做成一个网站吧，叫「吃在成都」。...

这个 prompt 对应的能力是：

- 支持社交媒体风格内容
- 识别 `📍`、`✅`、`🌟`、`#话题`
- 从种草笔记直接生成网站

这些样例都在 [evals/evals.json](/Users/yilink/Desktop/foodie-project/evals/evals.json)。

### 阶段 B：从“能看”变成“能互动”

这是这次项目里真实发生的产品迭代。

#### Prompt 5：评论区需求出现

> 之前有提到过，用户是可以在这个网站每一个餐厅下面去评论和留言的，现在好像没有看到可评论的地方啊

这一步把“静态内容网站”推进到“每个餐厅可互动”。

#### Prompt 6：要求评论状态外露

> 给评论还得登录，登录的话，除了 GitHub，其他的登录方式不行吗？另外，现在写完了评论，缩起来之后就看不见原来是否有用户评论过。怎么让这个留言在外面就能看到数字呢？

这一步明确了两个产品要求：

- 评论登录门槛不能太死
- 评论区不能藏在展开层里，外层要能看到数量

#### Prompt 7：按钮要更明显

> 第二，留言评论的button颜色不太突出，能不能让用户一看就觉得这里是可以点击的？

这一步推动了 CTA 强化，而不是只做“功能存在”。

#### Prompt 8：推荐入口位置前移

> 第三，最底部有新用户的推荐，但大部分人可能不会滑到最底下。你觉得重新换个位置的话，放在哪里更合适呢？

这一步推动了页面结构调整，把推荐入口从页尾前移。

### 阶段 C：从 GitHub 评论到 Firebase MVP

#### Prompt 9：多登录方式，选最快 MVP

> 多登录方式，这个麻烦吗？
>
> 1

这里的 `1` 表示选择“最快落地版本”，于是项目进入：

- Firebase Auth
- Google 登录评论
- Firestore 评论
- 保留旧方案回退

#### Prompt 10：开始做 MVP

> 开始做这个MVP吧

这是从讨论方案切到真正落地开发的明确开关。

### 阶段 D：从评论互动到共享点赞

#### Prompt 11：先做轻互动

> 把这个“大拇指点赞”MVP做进当前网站。

这一步先把点赞做进卡片交互。

#### Prompt 12：点赞必须是共享的

> 这个数字肯定是所有用户要共享呀，不能只是给我看。别的用户看到很多人点赞，他也想去嘛，大家都想看看

这一步很关键。它把“本地交互”升级成“真实产品信号”：

- 点赞数字必须所有用户共享
- 不能只是当前浏览器自己看见

于是点赞最终做成了：

- 共享计数
- 匿名身份
- 不要求手动登录

### 阶段 E：从能用到可上线

#### Prompt 13：我来配 Firebase，你来接

> firebase 协助Google登陆，花钱嘛？
>
> 那应该不太需要花钱，你能帮我操作吗？

这一步把项目从纯前端改造成真实后端接入。

后面又发生了多轮真实操作 prompt，例如：

- 询问 Analytics 怎么选
- 询问在哪里添加 Web App
- 询问在哪里开启登录方式
- 询问在哪里创建 Firestore
- 手动发布 Firestore 规则

这些 prompt 本质上展示的是：

- 用户不需要先懂 Firebase
- 只需要在关键节点回答和点击
- 其余都可以由代理式工作流协助完成

### 阶段 F：从“可用”到“手感好”

#### Prompt 14：第一次点赞没反应

> 为什么我点“觉得不错”，点了半天没反应，点了三四次才反应呢？

这一步推动了体验层优化：

- 点赞改为乐观更新
- 点击后立刻出现反馈
- 后台慢同步
- 失败再回滚

这不是“新功能”，但它非常适合展示：

一个真实项目不是只做功能，还会继续打磨交互手感。

## 3. 把这些 Prompt 压缩成一套对外展示的讲法

如果你要给别人 demo，可以把上面真实过程压缩成下面这 6 句。

### Showcase Prompt 1：从备忘录生成网站

> 我在备忘录里记了一堆美食和旅行地点，帮我做成一个网站，名字叫「Foodie」。这是我的内容：...

### Showcase Prompt 2：让网站支持互动

> 这个网站里，每个餐厅下面都要能评论和留言。

### Showcase Prompt 3：把评论入口做得更明显

> 评论按钮要更突出，而且外面就能看到这家店有没有人评论过。

### Showcase Prompt 4：把推荐入口前移

> 推荐新店的入口不要放在最底部，帮我放到用户更容易看到的位置。

### Showcase Prompt 5：把评论升级成 Google 登录

> 评论改成 Google 登录，阅读不用登录，发评论要登录。

### Showcase Prompt 6：加一个所有用户共享的点赞

> 每个餐厅加一个“大拇指点赞”，点赞数要所有用户共享，但不要要求用户手动登录。

只用这 6 句，已经足够把整个项目讲清楚。

## 4. 这类项目为什么适合做 Showcase

因为它天然满足三个层次：

### 层次 1：内容生产很简单

用户只要有：

- 备忘录
- 清单
- 小红书内容
- 旅行列表

就足够启动项目。

### 层次 2：网站生成很直接

这个 skill 的核心不是从零写网页，而是：

1. 把内容解析成 `data.js`
2. 把站点配置写成 `config.json`
3. 用模板生成 `index.html`

对应说明在 [SKILL.md](/Users/yilink/Desktop/foodie-project/SKILL.md)。

### 层次 3：后续迭代也可以继续用自然语言完成

比如：

- 加评论
- 换登录方式
- 强化按钮
- 调整版位
- 加共享点赞
- 修交互手感

这些都不是“重做项目”，而是自然语言持续迭代。

## 5. 这个网站是怎么被做成 skill 的

这个项目已经具备 skill 的基本结构，关键思路是：

### 输入

- 备忘录文本
- 文件内容
- 社交媒体复制文本

### 中间层

- 解析地点、地区、状态、标签
- 输出 `data.js`
- 输出 `config.json`

### 生成层

运行模板生成脚本：

```bash
python3 <skill-path>/scripts/setup.py config.json <skill-path>/assets/template.html index.html
```

### 输出

- `index.html`
- `data.js`
- 可直接部署到 GitHub Pages

也就是说，这个 skill 的本质不是“让模型凭空设计网站”，而是：

把“内容解析 + 配置注入 + 模板生成 + 部署”做成一条稳定工作流。

> 最后它就从一个静态页面，长成了一个真正能互动、能上线、能继续封装成 skill 的网站。
