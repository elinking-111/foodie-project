Outcome

这个项目最后做成了什么

不是 demo 图，而是已经跑在真实站点上的产品能力。

内容网站

Real Prompts

真实推动项目往前走的 Prompt

下面不是事后润色，而是这个项目里真正起作用的指令型句子。

1
从备忘录到网站
“我在手机备忘录里记了一堆北京和旅游的吃喝玩乐清单，帮我做成一个网站。”
解析地点自动分类生成网站
2
每家店都能评论
“用户是可以在这个网站每一个餐厅下面去评论和留言的。”
独立评论区从静态站变互动站
3
评论状态外露
“写完了评论，缩起来之后就看不见原来是否有用户评论过。怎么让这个留言在外面就能看到数字呢？”
外层计数交互反馈信息前置
4
CTA 要更明显
“留言评论的 button 颜色不太突出，能不能让用户一看就觉得这里是可以点击的？”
强化按钮提升可点感
5
最快版本的多登录方式
“多登录方式，这个麻烦吗？ 1”
Firebase MVPGoogle 评论登录保留回退
6
共享点赞不是本地计数
“这个数字肯定是所有用户要共享呀，不能只是给我看。”
共享计数匿名身份真实用户信号
7
让我能协助接 Firebase
“那应该不太需要花钱，你能帮我操作吗？”
Web App 配置Auth 开启Firestore 规则
8
继续打磨手感
“为什么我点‘觉得不错’，点了半天没反应，点了三四次才反应呢？”
乐观更新即时反馈体验修复

Skill Architecture

这个网站是怎么被做成一个 Skill 的

重点不是“模型直接写页面”，而是把内容解析、配置注入和模板生成变成稳定工作流。

Input

备忘录文本、文件、小红书复制内容、旅行清单，统统都可以作为原料。

Parse

抽取地点、地区、状态、标签，把内容规范化为结构化数据。

Build

生成 data.js 和 config.json，再用模板产出 index.html。

Deploy

推到 GitHub Pages，上线后继续用自然语言迭代评论、点赞和布局。

Artifacts

适合演示时一起打开的文件

这几个入口能把“prompt 到产品”的过程讲完整。

网站主页面 index.html

最终给用户看的真实站点。

模板文件 assets/template.html

真正把 skill 做稳定的关键，不是每次从零写网页。

Skill 说明 SKILL.md

把内容解析、生成和部署抽象成可复用工作流。

Prompt 样例 evals/evals.json

最早“把备忘录做成网站”的核心样例都在这里。

Showcase 文档 PROJECT_SHOWCASE.md

Firebase 接入 FIREBASE_SETUP.md

展示这个项目如何从静态站长成真实可互动产品。

一句话讲完这个项目

我不是先写完整需求文档再做网站，而是把备忘录内容贴进去，说“帮我做成一个网站”， 然后继续补几句“每家店可以评论”“评论按钮更明显”“加一个共享点赞”“评论改成 Google 登录”， 最后它就从一个清单页面，长成了一个真实能上线、能互动、还能沉淀成 skill 的产品。
